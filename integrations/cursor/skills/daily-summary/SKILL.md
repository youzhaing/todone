# Daily Summary — 每日工作收尾

配套工具：[todone](https://github.com/youzhaing/todone) 的 `td` 命令（Python 或 JS 实现都可）。

## WHAT

基于当天的 git commits、未提交改动、已完成 todo 和未完成 todo，替用户写一段**精简**的今日总结，并直接写入 `done.md` 今天那段下面的模板占位符。

## WHEN

用户在 Cursor chat 里说出触发词（见 frontmatter），意图是「结束今天的工作、留个总结到文件」时使用。**不要主动调用**，等用户明确要求。

## 工作流程

### 1. 跑 `td summary` 拿素材

在用户当前工作目录执行：

```bash
td summary
```

> 如果 `td` 命令找不到（例如在非登录 shell 里），改用绝对路径指向用户本机的 `td.py` 或 `td.js`，例如：
> `python3 /path/to/td.py summary`
> 或：
> `node /path/to/td.js summary`

这个命令会：

- 幂等地往 `done.md` 今天段落追加一段空模板（若已存在则不重复加）
- 在 stdout 打印一份「参考信息包」，包含：
  - `## Git repo` — 当前 repo 路径（或"不在 git 仓库内"）
  - `## 今日 commits` — `git log --since=midnight` 的 hash 和 subject
  - `## 未提交改动` — `git status -s` + `git diff --stat HEAD`
  - `## 当前未完成 todo` — `todo.md` 全部条目
  - `## 今日已完成 done` — 今天已经做掉的条目

### 2. 总结出 1 条今日核心产出，写进 done 列表

读完参考信息包 + 当前对话上下文后，用一句**简短（15-40 字）**的中文概括今天最核心的产出（做了什么事），然后执行：

```bash
td did "简短的一句话"
```

该命令会直接把这条追加到 `done.md` 今天段的 done 列表（总结模板之前），幂等：已有同名条目不重复加。

**写作规则**：

- **一句话**：不是流水账，是今天最值得被记下来的**一件事**。如果今天做了多件独立的事，可以调 `td did` 多次（每件一条，控制在 1-3 条以内）。
- **基于事实**：参考信息包或对话里真实发生的事，不编造。
- **动作 + 结果**：比如「给 td 加循环任务与日报命令」「重构 LTV 模型计算口径」。避免模板化的「完成今日工作」。
- **如果今天确实没有实质产出**（仅阅读/调研），可以跳过这一步，不强行加。

### 3. 写总结（三行精简版）

基于同一份事实写 **3 条**总结，覆盖 `done.md` 里的占位符：

```markdown
### 📝 今日总结
- 改了什么：[具体到文件/模块/功能，一句话，30-60 字]
- 为什么：[这些改动的目的/背景/解决的问题]
- 明天继续：[下一步具体行动，要可执行]
```

**写作规则**：

- **精简**：每行一句话，不罗列文件清单。
- **中文**：和用户的日常表达一致。
- **"明天继续"可执行**：写成具体动作（"接着写 X 的单元测试"），不要写空话。
- 如果某条确实没素材，就诚实写"（今日以调研/阅读为主）"之类，不硬凑。

### 4. 写入 `done.md`

用文件编辑工具（StrReplace）把模板里的三个占位符行替换成写好的内容。定位锚点用 `### 📝 今日总结` 确保只改今天那段。

替换示例（old_string）：

```markdown
### 📝 今日总结
- 改了什么：
- 为什么：
- 明天继续：
```

替换后（new_string）：

```markdown
### 📝 今日总结
- 改了什么：把 td.py 的总结命令改成幂等插入，沙盒测试通过
- 为什么：避免重复跑时往 done.md 反复塞模板，污染记录
- 明天继续：把 daily-summary skill 的流程补上自动追加"明日重点"到 todo.md
```

### 5.（可选）把"明天继续"同步到 todo 池

如果"明天继续"是具体可执行的条目，顺手执行：

```bash
td add "接着写 X 的单元测试"
```

让它出现在明天的 todo 里。如果是多条，分多次调用。

### 6. 回复用户

用一句话确认：「已记一条 done 并把今日总结写进 `done.md`（N 条明日重点已加到 todo 池）。」不要把总结原文大段复述，让用户自己去看文件。

## 边界

- **只处理今日**。如果用户问「总结上周」「总结这个月」，**拒绝并建议**用别的方式（比如 `td log 7` 拉完成记录、让 AI 基于 git log 自己归纳）。本 skill 的契约就是当天。
- **不做评价**。只描述事实（改了什么、为什么、下一步），不写"今天效率很高"之类主观判断。
- **模板占位符不存在就不硬改**。理论上 `td summary` 一定会确保模板存在；如果跑完后 `done.md` 里仍没有 `### 📝 今日总结`，说明脚本出错，应报告而不是手动瞎插。
